# Build Comparison Page — Design Spec

**Issue:** #6 (Plan 3) — Build comparison page
**Date:** 2026-04-09 (revised after Codex review)
**Scope:** `/compare` route, build list checkboxes, detail page "Compare with" link. Client-side diff from extended per-build JSON.

## Motivation

Plans 1–2 provide build list and detail views. Users need to compare two builds — see score deltas, structural differences (talents, weapons, blessings), synergy edge diffs, and breakpoint HTK comparisons. The library already has `diffBuilds()` but it uses `fs` and loads the full index; the website needs a browser-compatible diff layer operating on pre-computed data.

Client-side diff is chosen over pre-computation because it extends naturally to Plan 4 (GL import) — any two builds can be compared without pipeline changes.

## Routing & Entry Points

**Route:** `/compare?builds=slug1,slug2`

**Three entry points:**

1. **Build list page** — checkbox per row + "Compare" button. Select 2 builds → navigate to compare page.
2. **Build detail page** — "Compare with..." button in the header area → navigate to `/compare?builds={currentSlug},`.
3. **Direct URL** — `/compare?builds=slug1,slug2` is shareable.

**Fallback behavior:** If zero or one build slugs are provided, the compare page shows dropdown selectors to pick builds. The build list from `build-summaries.json` populates the dropdowns.

**Error handling:**
- Invalid slug: show "Build not found" with a dropdown to select a valid build
- One fetch fails: render the successful build's details, show error for the failed one
- Both builds are the same: show a notice "Comparing build with itself — all deltas will be zero" and still render
- Swap control: button to swap A ↔ B, which flips all delta signs

## Data Layer

### Data generation changes

The current per-build JSON has `slug`, `summary`, `scorecard`, `synergy`, `breakpoints` — but the structural diff needs slot assignments, talent IDs, weapon/blessing IDs, and curio perk IDs with display names. These exist in the raw build files but are not carried into the website payloads.

**New `structure` section** added to `BuildDetailData`, generated by `scripts/generate-data.ts` from the raw build data already loaded at generation time (line 54: `const build = JSON.parse(...)`):

```ts
interface BuildStructure {
  slots: {
    ability:  { id: string | null; name: string | null };
    blitz:    { id: string | null; name: string | null };
    aura:     { id: string | null; name: string | null };
    keystone: { id: string | null; name: string | null };
  };
  talents: Array<{ id: string; name: string }>;
  weapons: Array<{
    id: string;
    name: string;
    slot: string | null;
    family: string | null;
    blessings: Array<{ id: string | null; name: string }>;
  }>;
  curio_perks: Array<{ id: string | null; name: string }>;
}
```

**Why per-instance entries, not a flat ID→name map:** The same canonical ID can appear multiple times with different display labels (e.g., `shared.stat_node.toughness_boost` appears as both "Toughness Boost 4" and "Toughness Boost 5" in the same build; curio perk `shared.gadget_trait.gadget_toughness_increase` appears with different roll labels). A flat map destroys this multiplicity. Per-instance entries preserve it.

**Size impact:** ~40 entries per build, adds ~1-2KB per JSON file. Negligible relative to the ~2MB per-build payload.

**Generator changes** in `scripts/generate-data.ts`:
- After line 54 (`const build = JSON.parse(...)`), extract `structure` from the raw build
- Add `structure` to `buildDetailRecord()` and to the `BuildDetailData` output

**No library changes needed.** The `structure` section is purely a website data concern, extracted from the raw build during the existing generation pass.

### `compare.ts` — Pure diff functions

New module `website/src/lib/compare.ts` — pure functions on `BuildDetailData`, no `fs`, no library imports:

#### `computeScoreDeltas(a: BuildDetailData, b: BuildDetailData): CompareScoreDelta[]`

Delta per scoring dimension. `delta = b - a`. Uses `scorecard` section for composite/mechanical scores, `scorecard.qualitative` for qualitative scores. Dimension labels and max values come from a shared constant (see "Dimension metadata" section below).

#### `computeSetDiff(idsA: string[], idsB: string[]): CompareSetDiff`

```ts
interface CompareSetDiff {
  only_a: string[];
  only_b: string[];
  shared: string[];
}
```

**Multiset-aware:** Uses sorted count maps instead of plain sets. If A has `[x, x, y]` and B has `[x, y, y]`, result is `shared: [x, y]`, `only_a: [x]`, `only_b: [y]`. This correctly handles repeated stat nodes and curio perks with the same canonical ID.

For talents, weapons, blessings, curio perks.

#### `computeSlotDiff(a: BuildStructure, b: BuildStructure): CompareSlotDiff[]`

```ts
interface CompareSlotDiff {
  key: string;             // "ability", "blitz", "aura", "keystone"
  label: string;           // "Ability", "Blitz", "Aura", "Keystone"
  a: { id: string | null; name: string | null };
  b: { id: string | null; name: string | null };
  changed: boolean;
}
```

Operates on the new `structure.slots` data, not on `BuildSummary`.

#### `computeSynergyEdgeDiff(a: SynergyAnalysisDetail, b: SynergyAnalysisDetail): CompareEdgeDiff`

```ts
interface CompareEdgeDiff {
  only_a: SynergyEdgeDetail[];
  only_b: SynergyEdgeDetail[];
  shared: SynergyEdgeDetail[];
}

type SynergyEdgeKey = string; // type:families_sorted:selections_sorted
```

**Keying:** `type:sorted_families:sorted_selections` — includes families in the key to distinguish multiple stat_alignment edges for different families on the same selection pair. This prevents collapsing distinct edges (e.g., stat_alignment for toughness vs crit on the same pair).

Uses the full `SynergyEdgeDetail` objects as output (not just keys), so the UI can render type, strength, selections, and explanation without re-fetching.

#### `computeBreakpointDiff(a: BreakpointMatrixDetail, b: BreakpointMatrixDetail, scenario: string, difficulty: string): CompareBreakpointDelta[]`

```ts
interface CompareBreakpointDelta {
  breed_id: string;
  action_category: string; // "light", "heavy", "special" — normalized
  a_htk: number | null;
  b_htk: number | null;
  delta: number | null;    // b - a, negative = B kills faster
  a_weapon: string | null; // weapon that achieved a_htk
  b_weapon: string | null; // weapon that achieved b_htk
}
```

Compares HTK across both breakpoint matrices for the selected scenario/difficulty. Finds the best (lowest) HTK per breed/action-category across weapons. **Includes weapon attribution** so the user knows which weapon delivered the number — prevents misleading comparisons across melee/ranged profiles.

**Action category mapping:** The breakpoint matrix uses raw action types (`shoot_zoomed`, `weapon_special`, `push_followup`, etc.). The compare module normalizes these to `light`/`heavy`/`special` using a hardcoded map matching the library's `actionCategoryMap` in `damage-calculator.ts:1055`. This map is small (12 entries), stable, and explicitly copied rather than imported — it won't drift because the normalization is defined by the game, not by code.

#### Extractor helpers

```ts
function talentIds(detail: BuildDetailData): string[]
function weaponIds(detail: BuildDetailData): string[]
function blessingIds(detail: BuildDetailData): string[]
function curioPerkIds(detail: BuildDetailData): string[]
```

These extract from `detail.structure` (not from scorecard or synergy). The `structure` section is the authoritative source for diff-able entity IDs.

#### Dimension metadata

A shared constant (not duplicated from the detail page):

```ts
const DIMENSIONS = [
  { key: "composite_score", label: "Overall", max: 35 },
  { key: "perk_optimality", label: "Perks", max: 5 },
  { key: "curio_efficiency", label: "Curios", max: 5 },
  { key: "talent_coherence", label: "Talents", max: 5 },
  { key: "blessing_synergy", label: "Blessings", max: 5 },
  { key: "role_coverage", label: "Role", max: 5 },
  { key: "breakpoint_relevance", label: "Breakpoints", max: 5 },
  { key: "difficulty_scaling", label: "Scaling", max: 5 },
] as const;
```

The detail page's hardcoded dimension list (`+page.svelte:65`) should also reference this constant to avoid duplication. Refactor during implementation: extract to `$lib/dimensions.ts`.

## Page Structure

### Tabs

Five tabs: **Overview | Talents | Weapons | Synergy | Breakpoints**

### Overview Tab — Center-Delta Scorecard

Three-column layout:

- **Left panel:** Build A header (title, class badge, grade badge) + 8 score rows (composite + 7 dimensions), each showing `{label}: {score}/{max}`
- **Center column (narrow):** Delta values per row. Color-coded: green = B scores higher, red = A scores higher, gray = tied or both null. Delta format: `+N` / `-N` / `0`
- **Right panel:** Build B header + 8 score rows (mirroring left)

Below the scorecard: a **slot diff summary bar** showing ability/blitz/aura/keystone status (changed/same, with values).

Both build headers link to their respective detail pages (`/builds/{slug}`).

**Swap button** in the header area to swap A ↔ B.

**Cross-class badge:** Small "Cross-class comparison" indicator when the two builds are different classes. Note: structural and synergy diffs remain meaningful across classes; breakpoint comparisons are less directly comparable.

### Talents Tab — Three-Column Diff

- **Only in A** | **Shared** | **Only in B**
- Talent names from `structure.talents[].name`
- Below the three columns: **slot diff details** — shows ability, blitz, aura, keystone values for each build, with "changed" highlighted

### Weapons Tab — Three-Column Diff

- **Only in A** | **Shared** | **Only in B**
- For shared weapons: inline comparison of blessings showing which differ
- Weapon cards include slot label, family, and blessing summary
- Blessing names from `structure.weapons[].blessings[].name`

### Synergy Tab — Three-Column Edge Diff

- **Only in A** | **Shared** | **Only in B** — synergy edge cards (type, strength, selections, explanation)
- Below: anti-synergy comparison — anti-synergies unique to each build
- Below: coverage stats side-by-side (calc coverage, entities analyzed, build identity, coverage gaps)

### Breakpoints Tab — HTK Comparison Table

- Scenario/difficulty selectors (same as detail page)
- Table rows: per breed + action category combination
- Columns: Breed/Action | Build A HTK (weapon) | Delta | Build B HTK (weapon)
- Weapon attribution shown inline (e.g., "2 (Force Sword heavy)")
- Color coding: same as detail page's `htkCellClass` (green ≤1, yellow 2, red 3+), delta column uses green/red for direction
- Note below table: "Lower HTK is better. Green delta = Build B kills faster."

### Loading and Error States

- **Loading:** Show skeleton panels with spinner while fetching both build JSONs (~4-5MB total). Render each section as its data arrives (scorecard first, then structural, then synergy/breakpoints).
- **Partial failure:** If one build fetch fails, show error for that side and keep the other. Allow re-selecting the failed build.
- **Both loading:** Centered spinner with "Loading builds..." text.

### Mobile Layout

The three-column layouts collapse to stacked on narrow screens:
- Overview: single-column scorecard with inline deltas (A score → delta → B score per row)
- Talents/Weapons/Synergy: stacked sections (Only in A, then Shared, then Only in B) instead of side-by-side columns
- Breakpoints: full-width table with horizontal scroll (same as detail page)

Breakpoint: `md` (768px) — three columns above, stacked below.

## Build List Checkboxes

**On the build list page** (`+page.svelte`):

- Add a checkbox column on the left side of the table
- State: `selectedBuilds: Set<string>` tracking slugs, max 2 selections
- When 2 are selected, show a "Compare" button (positioned in the filter bar area)
- Clicking Compare navigates to `/compare?builds=slug1,slug2`
- When 0 or 1 are selected, the Compare button is disabled/hidden
- Prevent selecting more than 2 (disable remaining checkboxes when 2 are selected)
- Clear selection state when navigating away or when Compare is clicked

## Build Detail "Compare With"

**On the build detail page** (`/builds/[slug]/+page.svelte`):

- Add a "Compare with..." link in the header section, next to the "Back to builds" link
- Navigates to `/compare?builds={currentSlug},`
- The compare page detects the empty second slug and shows a dropdown pre-filtered to the current build's class (or all builds)

## Testing

### `compare.ts` unit tests

- `computeScoreDeltas`: same-build diff (all zeros), different builds (known deltas)
- `computeSetDiff`: empty sets, identical sets, disjoint sets, overlapping sets, multiset handling (repeated IDs)
- `computeSlotDiff`: same slots, different slots, null slots
- `computeSynergyEdgeDiff`: identical edges, unique edges per side, same pair different family
- `computeBreakpointDiff`: known HTK comparison for two real builds
- Extractor helpers: verify correct IDs extracted from `BuildDetailData.structure`

### Manual verification

- Load two builds via URL params, verify all tabs render correctly
- Cross-class comparison renders with badge
- Single build URL shows dropdown picker
- Build list checkboxes work: select 2 → Compare button appears → navigates correctly
- Detail page "Compare with..." link works
- Swap button flips deltas correctly
- Loading states render correctly
- Mobile: stacked layout renders correctly below 768px
- Same-build comparison shows zero-delta notice

## Deferred

- **Swap analysis on compare page** — showing what-if deltas if you swap a talent/weapon from the other build. This depends on the `recommend` module being browser-compatible, which requires more work.
- **Synergy graph visualization** — Svelte Flow rendering of edge diff. Deferred to v1.1 as noted in #6 brainstorm.
- **Curio perk dedicated tab** — curio perks are shown in the Overview slot diff section. Not enough data for a dedicated tab. The multiset-aware diff correctly shows duplicate perks.
