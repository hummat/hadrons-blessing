# BetterBots Integration Contract

> Date: 2026-03-15
> Status: Approved
> Issue: #4

## Problem

BetterBots is a Lua game mod with zero runtime access to Node.js. hadrons-blessing has structured build/entity data that BetterBots agents need when authoring Lua code (weapon template IDs, entity verification, build analysis). There is no defined handoff surface — agents cross-reference raw ground-truth JSON files ad hoc.

## Constraints

- No runtime coupling. BetterBots runs inside the Darktide game engine; it cannot call Node.js.
- Both repos are co-located on disk (`../hadrons-blessing` relative to BetterBots). This is the permanent working assumption.
- The consumer is an agent or human reading data to inform Lua code, not a build system importing artifacts.
- New exports must be justified by a concrete BetterBots use case, not added speculatively.

## Design

### Consumption model

**Primary (A):** BetterBots agent reads `../hadrons-blessing/data/exports/*.json` directly via filesystem.

**Secondary (C):** Agent runs CLI commands in the hadrons-blessing repo to regenerate exports or run ad-hoc queries (`npm run inspect`, `npm run resolve`). This is free since the CLI exists anyway.

No artifacts are duplicated into the BetterBots repo. No copy step, no sync mechanism.

### Export directory

```
data/exports/
  bot-weapon-recommendations.json   # per-class weapon picks for bot profiles
  README.md                         # contract documentation
```

Generated by CLI commands (e.g., `npm run export:bot-weapons`). Checked into git. Regenerated manually after build re-extraction or scorer changes.

### First artifact: `bot-weapon-recommendations.json`

Feeds BetterBots #45 (default class profiles). Pre-computes the non-trivial cross-reference of builds, bot-compatibility, and template IDs.

```json
{
  "generated_at": "2026-03-15T...",
  "schema_version": 1,
  "classes": {
    "veteran": {
      "melee": {
        "template_id": "combatsword_p2_m1",
        "display_name": "Catachan Mk VII \"Devil's Claw\" Sword",
        "canonical_entity_id": "shared.weapon.combatsword_p2_m1",
        "gestalt": "linesman",
        "source_builds": ["03-slinking-veteran"],
        "bot_notes": "Simple melee, no weakspot/dodge dependency"
      },
      "ranged": {
        "template_id": "autogun_p1_m1",
        "display_name": "Agripinaa Mk I Infantry Autogun",
        "canonical_entity_id": "shared.weapon.autogun_p1_m1",
        "gestalt": "killshot",
        "source_builds": [],
        "bot_notes": "Simple spray autogun — plasma/helbore excluded (BOT:AIM_DEPENDENT)"
      }
    },
    "zealot": { "..." : "..." },
    "psyker": { "..." : "..." },
    "ogryn": { "..." : "..." }
  }
}
```

The export targets the core 4 bot classes (veteran, zealot, psyker, ogryn). Arbites and hive_scum are excluded until BetterBots can validate them in-game (DLC-gated, see BetterBots #8).

Each class entry has `melee` and `ranged` with:
- `template_id` — player weapon template name (e.g., `combatsword_p2_m1`). This is the hadrons-blessing entity `internal_name`, which matches player item template strings. Note: vanilla *bot* profiles use different `bot_*`-prefixed IDs (e.g., `bot_combatsword_linesman_p1`); the `darktide_seven_*` misc profiles mix both player and bot template IDs. BetterBots #45 targets player templates to give bots real player-quality loadouts, using `MasterItems.get_item_or_fallback()` which accepts both formats.
- `display_name` — human-readable weapon name, sourced from build fixture `raw_label` values (ground-truth weapon entities have `ui_name: null` since display names are localized at runtime)
- `canonical_entity_id` — hadrons-blessing entity ID for cross-reference
- `gestalt` — part of the curated selection per weapon. Defaults are `"linesman"` for melee and `"killshot"` for ranged (matching BetterBots fallback behavior), but individual weapons may warrant different gestalts.
- `source_builds` — which of the 20 build fixtures informed this choice
- `bot_notes` — rationale for bot compatibility

#### Weapon selection criteria

Curated from the 20 canonical builds and qualitative bot-compatibility analysis documented in BetterBots' `docs/classes/meta-builds-research.md` (BOT:* tags like `BOT:AIM_DEPENDENT`, `BOT:NO_WEAKSPOT`, `BOT:NO_BLOCK_TIMING`). These tags are prose annotations in the BetterBots repo, not structured data in hadrons-blessing.

Selection principles:
1. Appears in at least one meta build for the class
2. No bot-incompatible characteristics (weakspot dependency, dodge-chaining, charged shot timing, block timing)
3. Simple attack patterns suitable for bot AI
4. Template ID verified against decompiled source weapon definitions

Selection is manual/curated, not algorithmic. The export command reads a curated mapping and enriches it with entity data, rather than trying to auto-derive bot compatibility.

### What stays CLI-only

| Need | Tool |
|------|------|
| Verify a talent internal name | `npm run inspect -- --id zealot.keystone.zealot_martyrdom` |
| Resolve a BetterBots weapon path | `npm run resolve -- --query "bot_lasgun_killshot" --context '{"kind":"weapon"}'` |
| Audit a build | `npm run audit -- scripts/builds/01-veteran-squad-leader.json` |
| Coverage report | `npm run coverage` |

These are ad-hoc queries, not stable reference data. No export needed.

### Future exports

Added only when a concrete BetterBots issue demands pre-computed data:
- Weapon family classification → when BetterBots #41 (weapon-aware ADS) is active
- Keystone behavioral annotations → when BetterBots #38 (talent-aware behavior) is active
- Tree topology data → when per-class talent path validation is needed

### Staleness policy

Exports are regenerated:
- After build re-extraction (new GL scrapes)
- After scorer or entity data changes that affect the export
- Before any BetterBots release that consumes the exports

The `generated_at` timestamp in each export enables staleness detection. No automated freshness enforcement — the agent checks the timestamp and regenerates if needed.

## Implementation

### File changes

| File | Change |
|------|--------|
| `scripts/export-bot-weapons.mjs` | New CLI script — reads builds + entities + curated weapon map, emits `data/exports/bot-weapon-recommendations.json` |
| `data/exports/bot-weapon-recommendations.json` | New generated artifact (checked in) |
| `data/exports/README.md` | New contract documentation |
| `package.json` | Add `export:bot-weapons` script |

### Curated weapon map

The script contains a hand-curated per-class weapon selection (template IDs chosen by examining meta builds and bot compatibility). The script validates that each template ID resolves to a known weapon entity in ground-truth, and enriches each entry with the `canonical_entity_id`. Display names are part of the curated map (sourced from build fixture `raw_label` values during curation, not looked up dynamically).

This is intentionally not algorithmic. Bot weapon suitability depends on qualitative factors (attack patterns, AI behavior quirks) that can't be derived from entity metadata alone.

## Testing

- Unit test: export script produces valid JSON matching the schema
- Unit test: all template IDs in the curated map resolve to existing weapon entities
- Integration: `npm run export:bot-weapons` runs without error and output is valid

## Non-goals

- No Lua code generation. The export is JSON; the BetterBots agent translates to Lua.
- No automated sync between repos. The cross-repo read is sufficient.
- No export versioning beyond `schema_version` in the JSON. Breaking changes bump the version.
